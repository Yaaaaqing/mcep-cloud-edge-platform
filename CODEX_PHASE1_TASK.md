# MCEP 前端第一轮收口修改任务

## 一、任务背景

当前项目已经具备首页、应用中心、数据中心、接入中心、边缘节点、运行监测、帮助文档和管理端等完整演示页面。本轮不是推翻重做，也不是接入正式后端，而是将现有 Demo 收口为一版可向领导、同事和合作方展示的“可信平台原型”。

请按“分析 → 修改 → 测试”的顺序执行。先阅读现有代码和数据结构，再实施修改；不要直接替换整个项目，不要新建平行项目。

## 二、本轮目标

本轮只处理以下四类问题：

1. 首页信息架构与视觉收口；
2. 全站数据口径和应用状态统一；
3. 明显交互 Bug、失效按钮和不准确文案；
4. 桌面端响应式、构建可复现和 SPA 部署兜底。

本轮不做：

- 正式后端、数据库、SSO、实时监控接口；
- 移动端完整适配；
- 大规模目录重构；
- 与本任务无关的页面重设计；
- 用虚假链接、虚假实时数据或未经确认的功能包装成已上线能力。

## 三、已提供资源

- `public/brand/mcep-logo-dark.png`：浅色导航栏使用的完整横版 Logo。

请直接使用上述资源，不要重新生成 Logo 或架构图，也不要在图片上再叠加重复文字。

# 四、具体修改要求

## A. 首页 `src/pages/HomePage.tsx`

### A1. 首屏结构

将首页首屏调整为明确的左右双栏：

- 左侧约 52%；
- 右侧约 48%；
- 右侧直接展示 `home-architecture.png`；
- 架构图应完整可见、比例不变、不得裁切重要文字；
- 不再使用当前由卡片和交叉细线组成的抽象 `ArchitectureDiagram` 作为首页主图，可保留组件供后续使用，但首页不要继续调用。

右侧建议采用白色或极浅蓝半透明容器，边框和阴影克制，不要再叠加复杂装饰。

### A2. 首页正式名称和标题

首屏必须直接出现平台正式名称：

`MCEP｜高端机床云边协同应用平台`

主标题固定为两行，不允许浏览器把“数据”等词拆开：

第一行：`连接工业软件与测试数据`

第二行：`协同云端平台与边缘节点`

建议将 `HomeContent.heroTitle` 保存为带换行符的字符串，并使用 `whitespace-pre-line` 渲染；管理端编辑时同样允许维护两行。

副标题改为：

`面向高端数控机床研发与测试场景，提供工业应用统一接入、数据资源管理、边缘节点协同和运行状态监测能力。`

标题桌面端不要超过约 60px；在 1366、1280、1024 宽度下不得出现单个词被拆行或标题顶到右侧主图。

### A3. 首屏按钮和辅助信息

保留两个按钮：

- `进入应用中心` → `/apps`
- `查看接入流程` → `/access`

保留公告，但视觉权重降低，不要压过平台名称和主标题。

底部“统一入口、数据协同、状态可追踪”可保留，但在 1024 宽度下允许换行，不能挤压布局。

### A4. 首页 KPI 改为 5 项并动态计算

首页 KPI 必须为：

1. 已接入应用：2
2. 联调中应用：1
3. 技术对接中：1
4. 数据资源：3
5. 边缘节点：规划建设中

禁止继续从 `statusSummary` 中写死数字。请根据实际数据动态计算：

- 已接入应用：`accessStatus === '已接入'`
- 联调中应用：`operationStatus === '联调中'`
- 技术对接中：`accessStatus === '技术对接中'`
- 数据资源：`resources.length`
- 边缘节点：当前没有正式在线节点时显示字符串 `规划建设中`

5 张卡片在 1440 及以上一行显示；1280/1024 下允许采用 3+2 或响应式布局，不能整体横向溢出。

### A5. 首页内容精简

首页只保留以下四个主体区块：

1. 首屏；
2. 5 个 KPI；
3. 重点建设成果；
4. `9.30 交付倒排`。

从首页移除：

- “核心能力”四卡片区；
- “五步完成规范化接入”区。

上述内容已经在接入中心或其他页面表达，不要在首页重复。相关数据常量如果其他页面仍使用则保留；如果没有使用，可以清理无效 import 和常量。

### A6. 重点建设成果

首页固定展示 3 个重点应用，状态表达必须为两个维度：

- 数控系统配置清单管理平台：`已接入｜业务运营中`
- 机床装配质量评价系统：`云端已部署｜联调中`
- 大连理工大学软件上云试点：`技术对接中｜待上云`

卡片按钮不得全部写成“进入应用”。无真实访问地址时统一使用“查看详情”。

### A7. 9.30 交付倒排

将当前泛化的 roadmap 替换为明确倒排节点：

1. 流程固化：06.30 前
2. 试点部署：07.31 前
3. 边缘节点管理：08.31 前
4. 数据展示验证：09.15 前
5. 最终交付：09.30 前

采用横向时间线或清晰的五节点结构。在 1024/1280 下可改为纵向时间线，但日期和节点名称必须完整显示。

## C. 应用数据模型与状态口径

涉及文件至少包括：

- `src/types/index.ts`
- `src/data/apps.ts`
- `src/components/AppCard.tsx`
- `src/pages/AppsPage.tsx`
- `src/pages/AdminPage.tsx`

### C1. 将单一 `status` 拆为两个字段

建议定义：

```ts
export type AppAccessStatus = '已接入' | '云端已部署' | '技术对接中' | '待上云' | '规划接入'
export type AppOperationStatus = '业务运营中' | '联调中' | '待上云' | '未运行'
```

`PlatformApp` 使用：

```ts
accessStatus: AppAccessStatus
operationStatus: AppOperationStatus
```

删除或完整迁移旧的单字段 `status`，不要让新旧字段长期并存造成口径混乱。

### C2. 初始应用状态

按以下口径设置：

- `app-001`：已接入｜业务运营中
- `app-002`：云端已部署｜联调中
- `app-003`：技术对接中｜待上云
- `app-004`：待上云｜未运行
- `app-005`：规划接入｜未运行
- `app-006`：已接入｜业务运营中

### C3. 应用卡片与详情

- 卡片同时显示接入状态和运行状态；
- 两个状态需有清晰但克制的视觉层级；
- 详情弹窗也分别显示两个状态；
- 有真实 `externalUrl` 时才显示“进入应用”，并安全地打开新窗口；
- 没有真实地址时显示“查看详情”；
- 删除现有 `https://example.com` 假链接；
- 不允许按钮名称暗示存在实际跳转，但点击后只打开详情。

### C4. 应用筛选

当前三个下拉框只显示“全部”，可读性不足。改为明确选项：

- 全部建设单位
- 全部应用类型
- 全部接入状态
- 全部运行状态

搜索框 + 4 个筛选器需做响应式布局；1024 宽度下可以分两行。

## I. 构建与部署可靠性

### I1. 固定依赖版本

`package.json` 中不得继续使用 `latest`。按照当前 `pnpm-lock.yaml` 已解析版本固定依赖，并更新 lockfile。至少包括：

- React 19.2.7
- React DOM 19.2.7
- React Router DOM 7.18.1
- Lucide React 1.24.0
- Recharts 3.9.2
- Vite 8.1.4
- `@vitejs/plugin-react` 6.0.3
- TypeScript 5.9.3

`@vitejs/plugin-react` 应放在 `devDependencies`。

### I2. SPA 路由刷新兜底

继续使用 `BrowserRouter`，不要改成 `HashRouter`。

新增：

- `vercel.json`：所有路径 rewrite 到 `/index.html`；
- `public/_redirects`：Netlify SPA fallback；
- `deploy/nginx-spa.conf.example`：提供 `try_files $uri $uri/ /index.html;` 示例。

确保 `/apps`、`/data`、`/access`、`/edge`、`/monitor`、`/admin` 直接访问和刷新时具备部署配置依据。
